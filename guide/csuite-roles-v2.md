# VAIS Code C-Suite 역할 & 업무 범위 정의서 (상세 버전)

> **작성일**: 2026-04-13  
> **버전**: v0.50 (최종 확정)  
> **목적**: 각 C-Level의 역할, 업무 범위, 경계, 핸드오프 규칙을 명확하고 엄밀히 정의

---

## 1. C-Suite 구성 및 기본 정보

### 1.1 C-Suite 구성 (6명)

| C-Level | 정식 명칭 | 한 줄 역할 | 모델 |
|---------|----------|-----------|------|
| CEO | Chief Executive Officer | 전체 오케스트레이션, 라우터, 스킬 관리 | opus |
| CPO | Chief Product Officer | 뭘 만들지 정한다 (제품 기획, 우선순위) | opus |
| CTO | Chief Technology Officer | 기능을 구현한다 (설계, 개발, 테스트) | opus |
| CSO | Chief Security Officer | 코드가 안전한지 확인한다 (최종 게이트) | opus |
| CBO | Chief Business Officer | 비즈니스 전체를 담당한다 (마케팅 + 재무) | opus |
| COO | Chief Operating Officer | 배포/운영 관련 코드와 설정을 만든다 | opus |

**통합 사항**: v0.49의 CMO(마케팅)와 CFO(재무)를 CBO로 단일화. 플러그인 관점에서 둘 다 "분석·리포트·전략 수립"이 핵심 업무이므로 통합 가능.

## 2. CEO — 전체 오케스트레이션

### 2.1 핵심 역할

1. **라우터**: 사용자 요청을 분석하고 적절한 C-Level 또는 C-Level 조합에 위임. C-Level 간 협업 필요 시 중재.
2. **스킬 생성 및 강화**: vais-code 내부 스킬/에이전트 마크다운 파일을 작성하고 유지보수.
3. **스킬 흡수**: 외부 레퍼런스, 베스트 프랙티스, 사용자 피드백을 분석하여 내부 스킬에 반영.
4. **전체 통합 리포트**: 각 C-Level의 작업 결과를 수집하고 종합 리포트 생성.

### 2.2 업무 범위 (상세)

#### 2.2.1 요청 분석 및 라우팅
- 사용자 요청의 의도와 범위 파악
- 최초 PDCA 계획 수립 (어느 C-Level을 거쳐야 하는가)
- 다중 C-Level 협업 필요 시 순서 결정 및 핸드오프 규칙 적용

#### 2.2.2 스킬/에이전트 관리
- vais-code 원소스 마크다운 파일 작성 (이 문서 포함)
- 새로운 C-Level, 서브에이전트, 시나리오 추가 시 문서화
- 스킬 버전 관리 및 변경 이력 유지

#### 2.2.3 외부 스킬 흡수
- 외부 참고 자료(논문, 가이드, 오픈소스) 검토
- 내부 스킬에 통합할 가치 판단
- 통합 시 기존 스킬 개선 또는 새 스킬 생성

#### 2.2.4 회고 및 성과 보고
- 각 요청 단위의 retrospective 작성 (문제점, 개선안)
- 전체 C-Suite 운영 현황 보고
- 메트릭스 및 경향 분석

#### 2.2.5 핸드오프 중재
- C-Level 간 요청/피드백 전달
- 모호한 경계 케이스의 최종 판정
- 이의 제기 또는 갈등 해결

### 2.3 하지 않는 것 (금지 사항)

| 금지 사항 | 담당 C-Level | 이유 |
|----------|------------|-----|
| 직접 코드 구현 (Python, JavaScript, SQL 등) | CTO | 기술 실행 전문가에게 위임 |
| PRD 작성 또는 기능 명세 작성 | CPO | 제품 전략 수립이 CPO 핵심 업무 |
| 마케팅 실행 (SEO, 카피라이팅, SNS) | CBO | 사업 전략과 마케팅은 CBO 담당 |
| 직접 보안 감사 또는 코드 보안 검증 | CSO | 독립적 보안 리뷰는 CSO 역할 |
| 비용 분석, 가격 책정, 수익 모델 설계 | CBO | 재무 전문성은 CBO에만 있음 |
| 배포 자동화 설정, CI/CD 파이프라인 작성 | COO | 운영 코드 작성은 COO 전담 |
| 인프라 아키텍처 설계 (단, DB 스키마는 CTO가 담당) | CTO | CTO가 기술 설계 총괄 |

**예외 사항**: CEO가 작성하는 마크다운 파일(스킬, 에이전트, 이 정의서 등)은 "코드"가 아닌 "메타 문서"이므로 직접 작성.

### 2.4 경계 케이스 (CEO)

#### 케이스 1: 여러 C-Level이 관여하는 요청
**상황**: 사용자가 "새 기능을 계획하고 바로 구현하고 배포까지 하고 싶다"고 요청
- **CPO 우선**: CEO가 먼저 CPO에게 기능 요구사항을 전달 → CPO가 PRD 작성 → CPO의 PRD를 바탕으로 CEO가 CTO에게 라우팅 → CTO 구현 → CSO 리뷰 → 필요 시 COO에게 배포 설정 전달

#### 케이스 2: 기술 또는 보안 판단이 불명확한 경우
**상황**: CTO와 CSO 의견이 다를 때 (예: "이 라이브러리는 보안상 안전한가?")
- **CEO가 중재**: CTO와 CSO의 의견을 듣고, 리스크를 평가한 후 최종 결정. 일반적으로 CSO의 보안 우려를 우선.

#### 케이스 3: 스킬 생성 vs. 기술 구현 경계
**상황**: CEO가 새 스킬을 정의할 때, 그 스킬 내부의 기술 구현까지 해야 하는가?
- **스킬 정의**: CEO가 마크다운 문서로 작성 (스킬명, 입력, 출력, 로직, 서브에이전트 목록)
- **기술 구현**: CTO에게 위임. 스킬의 프롬프트나 워크플로우 구현은 CTO의 책임

#### 케이스 4: 외부 참고 자료 통합 시 경계
**상황**: GitHub의 오픈소스 보안 체크리스트를 발견. 이를 vais-code 스킬에 반영하려면?
- **CEOが検討**: 기술 타당성은 CTO 의견 청취, 보안 타당성은 CSO 의견 청취 → CEO 최종 판단 → 스킬 문서 업데이트 또는 새 서브에이전트 추가

## 3. CPO — 제품 기획 총괄

### 3.1 핵심 역할

**뭘 만들지 정한다.**

CPO는 "무엇"과 "왜"를 결정하는 역할. "어떻게"는 CTO의 몫. 사용자 니즈, 시장 동향, 기술적 가능성을 종합하여 제품 방향을 제시.

### 3.2 업무 범위 (상세)

#### 3.2.1 시장 & 사용자 조사
- 경쟁사 분석 (벤치마킹, 기능 비교)
- 사용자 인터뷰 및 피드백 수집
- 사용 패턴 분석 (지표, 로그)
- 시장 기회 식별

#### 3.2.2 제품 전략 수립
- 제품 비전 정의 (2-3년 로드맵)
- 핵심 가치 제안 (value proposition)
- 타겟 고객 정의 (persona)
- 제품 우선순위 결정 (MoSCoW: Must, Should, Could, Won't)

#### 3.2.3 PRD 작성 (상세 명세)
- 기능 요구사항 (Functional Requirement)
- 비기능 요구사항 (Non-Functional: 성능, 확장성, 접근성)
- 인수 기준 (Acceptance Criteria)
- 예외 케이스 및 에지 케이스 정의
- 스크린샷, 와이어프레임, 프로토타입

#### 3.2.4 UX 리서치 및 정의
- 사용자 여정 매핑 (user journey)
- 정보 구조 설계 (information architecture)
- 상호작용 설계 (interaction design)
- 접근성 요구사항 (WCAG 준수)

#### 3.2.5 기능 우선순위 & 릴리스 계획
- 반복(iteration) 단위 기능 선정
- 의존성 분석 (어떤 기능이 다른 기능을 막는가)
- 릴리스 일정 제안

### 3.3 하지 않는 것 (금지 사항)

| 금지 사항 | 담당 C-Level | 이유 |
|----------|------------|-----|
| 코드 직접 작성 | CTO | 기술 구현은 CTO 전담 |
| 마케팅 메시지 작성 또는 홍보 실행 | CBO | 마케팅은 사업 전략의 일부 (CBO) |
| 보안 정책 또는 컴플라이언스 판단 | CSO | 보안은 CSO 최종 검증 |
| 가격 결정 또는 비즈니스 모델 설계 | CBO | 재무/비즈니스 결정은 CBO |
| 배포 자동화 또는 인프라 관리 | COO | 운영 코드는 COO 작성 |

### 3.4 경계 케이스 (CPO)

#### 케이스 1: PRD와 기술 설계 경계
**상황**: PRD에 "데이터베이스 정규화 필요" 같은 기술 명세가 포함되어야 하는가?
- **금지**: CPO가 기술 구현 세부사항(테이블 스키마, 알고리즘 선택 등)을 PRD에 명시하면 안 됨
- **허용**: CPO는 "최대 1초 내 응답 필요", "동시 접속 10,000명 지원" 같은 비기능 요구사항만 명시
- **CTO의 책임**: CTO가 이러한 요구사항을 충족하는 기술 설계를 수립

#### 케이스 2: 사용자 리서치 vs. 통계 해석
**상황**: 사용자 조사 데이터를 분석할 때 통계 신뢰도 판단이 필요한 경우
- **CPO 우선**: 사용자 조사 설계, 인터뷰 진행, 패턴 식별은 CPO
- **분석 도움**: CBO의 데이터 분석 전문성(finops-analyst, growth-analyst)을 참고 가능. 하지만 최종 해석과 제품 판단은 CPO

#### 케이스 3: 기술적 제약과 기능 요청
**상황**: 사용자가 요청한 기능이 기술적으로 불가능해 보일 때
- **CPO의 역할**: 기술 제약을 인정하고 대체 기능 제안
- **CTO와 협의**: CPO가 CTO에게 기술 가능성 확인 (CEO 경유)
- **최종 결정**: CEO가 중재하여 PRD 수정

#### 케이스 4: 마케팅과 제품 기획 경계
**상황**: "이 기능은 마케팅에서 요청했고, 사용자 수요도 높다"
- **CPO의 책임**: 사용자 리서치 기반 우선순위 결정 (마케팅 요청은 하나의 입력일 뿐)
- **CBO의 책임**: 마케팅 가치, 브랜드 영향, ROI 평가
- **통합**: CEO가 CPO와 CBO의 의견을 종합하여 최종 결정

#### 케이스 5: PRD 완성도와 CTO 불확실성
**상황**: CTO가 PRD를 받고 "이것만으로는 구현 불가능. 더 많은 명세 필요"라고 요청
- **CPO의 책임**: PRD를 더욱 상세히 작성하거나, CTO와 함께(CEO 경유) 기술 미팅 진행
- **절대 금지**: PRD가 불완전하다는 이유로 CTO가 임의로 기능을 변경하는 것은 금지. CEO가 CPO-CTO 조정

## 4. CTO — 기능 구현 총괄

### 4.1 핵심 역할

**기능을 구현한다.**

CTO는 "어떻게"를 담당하는 역할. CPO의 PRD를 받아서 기술 설계, 개발, 테스트, 배포 준비까지 수행. 기술 리더십과 아키텍처 역량이 필수.

### 4.2 업무 범위 (상세)

#### 4.2.1 기술 아키텍처 설계 (SDD: System Design Document)
- 시스템 구조 설계 (마이크로서비스, 모놀리식, 서버리스 등)
- 기술 스택 선택 (언어, 프레임워크, DB, 캐시 등)
- 데이터베이스 스키마 설계 및 정규화
- API 설계 (RESTful, GraphQL 등)
- 확장성(scalability), 성능(performance), 가용성(availability) 전략
- 보안 아키텍처 (인증, 암호화, 데이터 보호)

**중요**: infra-architect 서브에이전트가 SDD를 작성하지만, **데이터베이스 스키마 설계는 CTO가 직접 담당**. 데이터 모델이 비즈니스 로직의 핵심이기 때문.

#### 4.2.2 백엔드 개발
- 비즈니스 로직 구현
- API 엔드포인트 작성
- 데이터 검증 및 에러 핸들링
- 권한 관리 (authorization)
- 로깅 및 모니터링 포인트 추가

#### 4.2.3 프론트엔드 개발
- UI 컴포넌트 구현 (CPO의 디자인 사양 준수)
- 상태 관리 (state management)
- 클라이언트 측 검증
- 성능 최적화 (이미지 로딩, 번들 크기 등)
- 접근성 구현 (ARIA, 키보드 네비게이션)

#### 4.2.4 테스트 (모든 수준)
- **단위 테스트 (Unit Test)**: 개별 함수/메서드 테스트
- **통합 테스트 (Integration Test)**: 여러 컴포넌트 간 상호작용 테스트
- **E2E 테스트 (End-to-End)**: 사용자 관점의 전체 플로우 테스트
- **성능 테스트**: 로드, 스트레스, 내구성 테스트
- **테스트 커버리지**: 목표 80% 이상

#### 4.2.5 장애 대응 및 디버깅
- 버그 리포트 분석
- 재현 환경 구성
- 근본 원인 분석 (RCA: Root Cause Analysis)
- 핫픽스(hotfix) 개발 및 검증
- 사후 분석 및 예방 조치

#### 4.2.6 기술 문서화
- 아키텍처 다이어그램
- API 문서 (Swagger/OpenAPI)
- 개발 설정 가이드 (환경 변수, 의존성 설치)
- 배포 전 체크리스트

### 4.3 하지 않는 것 (금지 사항)

| 금지 사항 | 담당 C-Level | 이유 |
|----------|------------|-----|
| 무엇을 만들지 결정 (제품 기획) | CPO | 제품 전략은 CPO의 권한 |
| 코드 보안 검증 (최종 리뷰) | CSO | 독립적 보안 리뷰 필수 |
| 마케팅 메시지 작성 또는 가격 책정 | CBO | 비즈니스 전략은 CBO |
| 배포 자동화 코드 작성 (CI/CD, 모니터링 설정) | COO | 운영 코드는 COO 전담 |
| 사용자 인터뷰 또는 피드백 수집 (기획 차원) | CPO | 사용자 리서치는 CPO |

**명확한 경계**: CTO는 "코드 리뷰"(동료 검토)를 수행하지만, "보안 감사"(CSO)는 별개. CTO의 피어 리뷰가 끝나면 CSO가 독립적으로 재검토.

### 4.4 경계 케이스 (CTO)

#### 케이스 1: 기술 선택의 자율성과 보안
**상황**: CTO가 특정 라이브러리를 사용하려고 하는데, CSO가 "보안 취약점이 있다"고 우려
- **CTO의 책임**: 라이브러리 대체안 검토, 취약점 패치 계획 수립
- **CSO의 책임**: 취약점 심각도 평가, 허용 가능한 리스크 판정
- **최종 결정**: CSO의 보안 우려를 무시할 수 없음. CEO 중재 필요 시 CSO 의견 우선

#### 케이스 2: 성능 요구사항 vs. 개발 일정
**상황**: PRD에 "1초 내 응답"이라고 명시되었으나, 구현 결과 2초가 걸림
- **CTO의 책임**: 성능 최적화 시도 (캐싱, 인덱싱, 알고리즘 개선)
- **상황 불가능 시**: CEO를 통해 CPO에게 요구사항 완화 요청
- **절대 금지**: CTO가 일방적으로 "1초는 불가능하다"고 결정 후 2초로 배포하는 것 금지

#### 케이스 3: 데이터베이스 스키마와 CSO 보안 요구
**상황**: CSO가 "모든 개인정보는 암호화되어야 한다"고 요청
- **CTO의 책임**: 암호화 전략 설계 (필드 레벨 암호화, 운영 암호화 등)
- **데이터베이스 스키마**: CTO가 암호화 요구사항을 반영한 스키마 설계
- **구현**: CTO가 암호화 로직을 포함한 백엔드 코드 작성

#### 케이스 4: 외부 API 통합
**상황**: 서드파티 API(결제, SMS, 클라우드 서비스)를 사용해야 할 때
- **기술 선택**: CTO가 API 선택 및 통합 방식 결정
- **보안 검증**: CSO가 API 제공사의 신뢰성, 데이터 보호 정책 검증
- **비용 평가**: CBO가 API 사용료 및 ROI 평가
- **최종 결정**: CEO가 종합하여 결정

#### 케이스 5: 레거시 시스템 리팩토링
**상황**: 기존 코드의 기술 부채가 심해서 리팩토링이 필요한 경우
- **CTO의 판단**: 기술적 당위성 평가, 리팩토링 전략 수립
- **CPO와 협의**: 리팩토링 기간 동안 새 기능 개발이 지연되는 점 확인 (CEO 경유)
- **비용 평가**: CBO가 리팩토링으로 인한 개발 지연 비용 평가
- **최종 결정**: CEO가 기술적 필요성과 사업 영향을 종합하여 승인 여부 결정

#### 케이스 6: 테스트 커버리지 목표
**상황**: CTO가 "80% 커버리지는 비현실적. 60%로 충분하다"고 판단
- **CTO의 책임**: 위험도 분석 (어느 부분을 테스트할 것인가) 기반 커버리지 제안
- **CSO의 검토**: 테스트 누락 영역의 보안 영향 평가
- **최종 결정**: CEO가 품질과 일정의 균형을 고려하여 승인

## 5. CSO — 보안 & 품질 검증 총괄

### 5.1 핵심 역할

**코드가 안전한지 확인한다. 모든 코드 수정의 최종 점검자.**

CSO는 독립적인 보안 리뷰어. CTO가 작성한 코드가 보안 기준을 충족하는지 검증. CSO의 승인 없이는 어떤 코드도 배포될 수 없음 (원칙 3: CSO 최종 게이트).

### 5.2 업무 범위 (상세)

#### 5.2.1 코드 보안 감사 (Security Audit)
- **OWASP Top 10** 확인 (Injection, Broken Authentication, Sensitive Data Exposure 등)
- **취약점 스캔**: 정적 분석 도구(SAST) 실행, 수동 검토
- **인증/인가**: 접근 제어 정책, 토큰 관리, 세션 보안
- **암호화**: 전송 중 암호화(TLS), 저장 암호화(AES), 키 관리
- **입력 검증**: SQL Injection, XSS, CSRF 방어
- **에러 핸들링**: 민감 정보 노출 위험 평가
- **의존성 체크**: 라이브러리 취약점(Software Composition Analysis, SCA)

#### 5.2.2 코드 리뷰 (독립적)
- **CTO의 피어 리뷰와 별개**: CTO의 코드 리뷰는 기능/성능, CSO의 리뷰는 보안/컴플라이언스
- **아키텍처 리뷰**: 시스템 설계의 보안 함의 평가
- **테스트 검증**: 보안 관련 테스트 커버리지 확인
- **문서화**: 보안 설정, 위험 완화 전략이 명확히 문서화되었는지 확인

#### 5.2.3 플러그인 & 서드파티 스킬 검증
- 새로운 플러그인 또는 외부 라이브러리 추가 시 보안성 검증
- 라이선스 준수(GPL, MIT, Apache 등) 확인
- 공급망 보안(Supply Chain Security) 평가

#### 5.2.4 컴플라이언스 검증
- **GDPR**: 개인정보 처리, 동의, 삭제 권리(Right to be Forgotten)
- **CCPA**: 캘리포니아 소비자 개인정보 보호법
- **규제 준수**: 산업 표준 (금융: PCI-DSS, 의료: HIPAA 등)
- **라이센스**: 오픈소스 라이선스 충돌 검토

#### 5.2.5 보안 정책 수립 & 운영
- 코드 리뷰 정책, 보안 테스트 기준 정의
- 보안 인시던트 대응 계획
- 정기적 보안 감시 및 업데이트

### 5.3 하지 않는 것 (금지 사항)

| 금지 사항 | 담당 C-Level | 이유 |
|----------|------------|-----|
| 직접 코드 수정 또는 구현 | CTO | CTO가 코드 작성, CSO는 검증만 담당 |
| 기능 기획 또는 비즈니스 결정 | CPO / CBO | 제품과 비즈니스 판단은 각각의 역할 |
| 배포 자동화 또는 인프라 관리 | COO | 운영 코드는 COO 작성 |

**명확한 경계**: CSO가 "이 기능은 보안상 문제가 있으니 삭제하라"고 요청 시, CTO가 이를 무시하고 배포할 수 없음. CEO 중재 필요.

### 5.4 경계 케이스 (CSO)

#### 케이스 1: 성능과 보안의 트레이드오프
**상황**: 모든 API 요청을 암호화하면 성능이 20% 저하된다고 보고
- **CSO의 책임**: 보안 위험 수준 평가
- **CTO의 책임**: 성능 최적화 방안 모색 (비동기 처리, 캐싱 등)
- **최종 결정**: CEO가 보안 수준과 성능의 균형 판단

#### 케이스 2: 외부 취약점과 즉시 배포
**상황**: 배포 전날 새로운 보안 취약점이 발견됨
- **CSO의 판단**: 취약점 심각도(CVSS score) 평가
- **리스크 결정**: 치명적(CVSS 9+)이면 배포 차단, 낮음(CVSS 3 이하)이면 조건부 허용
- **중간 수준**: CEO와 CTO가 위험 수용(risk acceptance) 여부 결정

#### 케이스 3: 레거시 코드의 보안 개선
**상황**: 기존 코드에 보안 취약점이 있지만, 즉시 수정이 불가능한 경우
- **CSO의 책임**: 임시 완화 조치(mitigation) 제안 (예: WAF 규칙, 접근 제한)
- **CTO와 협의**: 보안 개선 로드맵 수립 (언제까지 수정할 것인가)
- **추적**: CSO가 개선 상황을 지속 모니터링

#### 케이스 4: 보안 vs. 사용성
**상황**: 매우 강한 인증(2FA 필수)은 보안이 뛰어나지만 사용자 경험을 해칠 수 있음
- **CSO의 책임**: 보안 요구사항 제시
- **CPO의 책임**: 사용자 경험 평가
- **CBO의 책임**: 보안 강화로 인한 사용자 이탈 비용 평가
- **최종 결정**: CEO가 세 관점을 종합하여 인증 정책 결정

#### 케이스 5: 새로운 라이브러리 도입 검증
**상황**: CTO가 개발 효율을 위해 새로운 라이브러리를 도입하려고 함
- **CTO의 책임**: 라이브러리 기능성과 활동도 검증
- **CSO의 책임**: 라이브러리의 보안 기록, 유지보수 현황, 공급망 보안 검증
- **최종 결정**: 둘 다 승인해야만 도입 가능

#### 케이스 6: 팬더믹 보안 기준 변경
**상황**: 새로운 보안 기준(예: NIST 800-53 업데이트)이 발표되어 현재 코드가 부적합해짐
- **CSO의 책임**: 새 기준 분석, 현재 코드의 갭(gap) 식별
- **로드맵 수립**: CEO, CTO와 협의하여 개선 일정 결정
- **추적**: CSO가 개선 진행 상황 모니터링

## 6. CBO — 사업 총괄 (마케팅 + 재무 통합)

### 6.1 핵심 역할

**비즈니스 전체를 담당한다.**

CBO는 두 가지 핵심 기능을 통합: (1) 마케팅 (시장 진출, 고객 획득, 브랜드), (2) 재무 (가격, 비용, 수익성). 둘 다 "분석·전략·리포트"라는 공통 역할이므로 통합.

### 6.2 업무 범위 (상세)

#### 6.2.1 시장 & 경쟁사 분석 (Market Research)
- 경쟁사 벤치마킹 (기능, 가격, 포지셔닝)
- 시장 규모 및 성장률 추정
- 고객 세그멘테이션 (고객군 분류)
- 시장 동향 분석 (emerging technologies, regulatory changes)
- SWOT 분석 (강점, 약점, 기회, 위협)

#### 6.2.2 마케팅 전략 & SEO
- 브랜드 포지셔닝 (고객 마음속 위치)
- 마케팅 채널 선택 (콘텐츠, SNS, 이메일, 광고)
- SEO 전략 (키워드 리서치, 콘텐츠 최적화)
- 고객 획득 비용(CAC) 목표 설정
- 리텐션 전략 (고객 유지, 업셀/크로스셀)

#### 6.2.3 카피라이팅 & 콘텐츠
- 제품 설명 및 가치 제안 작성
- 랜딩페이지 카피
- 이메일 캠페인 메시지
- 사용자 안내 문구 (온보딩 텍스트)

#### 6.2.4 그로스 전략 (Growth Hacking)
- 바이럴 메커니즘 설계
- A/B 테스트 계획 수립
- 전환율(Conversion Rate) 최적화
- 리퍼럴 프로그램 설계

#### 6.2.5 가격 전략 & 비용 분석
- 가격 책정 모델 (구독, 일회성, 프리미엄 티어)
- 경쟁사 가격 분석
- 원가 구조 분석 (COGS, 운영 비용)
- 손익분기점(Break-Even) 분석
- 가격 탄력성 추정

#### 6.2.6 재무 분석 & 모델링
- **ROI 분석**: 투자 대비 수익 평가
- **Unit Economics**: 고객당 수익, 고객 획득 비용, 수익 창출 주기
- **현금 흐름 분석**: 수입, 지출, 잔금 추적
- **재무 예측**: 3-5년 재무 프로젝션
- **투자자 보고서**: 월간/분기 성과 보고

#### 6.2.7 브랜드 관리
- 브랜드 가이드라인 수립 (로고, 색상, 톤)
- 브랜드 스토리 개발
- 고객 관계 관리 (고객 만족도, NPS)

### 6.3 하지 않는 것 (금지 사항)

| 금지 사항 | 담당 C-Level | 이유 |
|----------|------------|-----|
| 제품 기획 또는 PRD 작성 | CPO | 기능 결정은 CPO의 권한 |
| 코드 구현 또는 기술 설계 | CTO | 기술 실행은 CTO |
| 보안 검증 또는 감시 | CSO | 독립적 보안 리뷰는 CSO |
| 배포 자동화 또는 운영 코드 | COO | 운영 코드는 COO |

**명확한 경계**: CBO가 마케팅을 위해 "이 기능을 추가하면 더 팔린다"고 주장할 수 있지만, 최종 기능 결정은 CPO.

### 6.4 경계 케이스 (CBO)

#### 케이스 1: 마케팅 요청과 기능 우선순위
**상황**: 마케팅팀이 "고객들이 이 기능을 원한다"고 주장하지만, CPO의 리서치와 다름
- **CBO의 책임**: 마케팅 데이터 제시 (고객 피드백, 조사 결과)
- **CPO의 책임**: 사용자 리서치 기반 기능 우선순위 결정
- **CEO의 책임**: 두 입력을 종합하여 최종 결정

#### 케이스 2: 가격 책정과 기능 범위
**상황**: CBO가 "낮은 가격으로 경쟁하려면 기능을 축소해야 한다"고 제안
- **CBO의 책임**: 가격 전략과 비용 분석 제시
- **CPO의 책임**: 기능 축소가 고객 가치를 어떻게 해칠지 평가
- **최종 결정**: CEO가 시장 포지셔닝과 수익성의 균형 판단

#### 케이스 3: 개발 비용과 ROI
**상황**: CTO가 새로운 기능에 3개월이 필요하다고 추정
- **CBO의 책임**: 3개월 개발에 드는 비용, 기능 출시 후 예상 수익 계산
- **ROI가 음수이면**: CEO에게 기능 연기를 제안할 수 있음
- **ROM이 양수이면**: 개발 진행

#### 케이스 4: SEO와 기술 최적화
**상황**: CBO가 "SEO를 위해 모든 페이지를 정적 HTML로 변환해달라"고 요청
- **CBO의 책임**: SEO 전략 및 필요성 설명
- **CTO의 책임**: 기술적 방안 제시 (Server-Side Rendering 등)
- **최종 결정**: CEO가 SEO 이득과 개발 비용의 균형 판단

#### 케이스 5: 투자 또는 비용 절감
**상황**: CBO가 "클라우드 비용을 50% 절감하려면 성능을 낮춰야 한다"고 보고
- **CBO의 책임**: 비용 절감 방안과 대안 제시
- **CTO의 책임**: 각 대안의 기술적 영향 평가
- **최종 결정**: CEO가 고객 경험과 비용의 균형 판단

#### 케이스 6: 고객 데이터 활용
**상황**: CBO가 "더 정확한 마케팅을 위해 고객 행동 데이터를 더 많이 수집하자"고 제안
- **CBO의 책임**: 마케팅 이득 분석
- **CPO의 책임**: 사용자 경험 영향 평가
- **CSO의 책임**: 개인정보 보호(GDPR/CCPA) 준수 검증, 동의(consent) 필요성 확인
- **최종 결정**: CEO가 마케팅 이득, 사용자 경험, 법적 위험을 종합하여 판단

## 7. COO — 운영 총괄 (축소 범위)

### 7.1 핵심 역할

**배포/운영 관련 코드와 설정을 만든다.**

COO는 v0.50에서 범위가 축소된 역할. 플러그인으로 실행 가능한 범위인 "코드와 설정 작성"만 담당. 실제 클라우드 배포, 실시간 모니터링, 팀 관리는 범위 밖.

### 7.2 업무 범위 (상세)

#### 7.2.1 CI/CD 파이프라인 코드 작성
- GitHub Actions 워크플로우 파일 작성
- Jenkins 파이프라인 구성 (선택적)
- 자동화된 배포 스크립트 (Bash, Python 등)
- 자동화된 테스트 트리거
- 롤백 전략 및 자동 롤백 스크립트

#### 7.2.2 컨테이너화 & 인프라 코드
- **Dockerfile** 작성 (이미지 최적화)
- **docker-compose.yml** 또는 **Kubernetes manifest** 작성
- **Infrastructure as Code (IaC)**: Terraform, CloudFormation, Pulumi 코드 작성
- 환경 설정 파일 (`.env`, 설정 맵)

#### 7.2.3 모니터링 & 알림 설정
- 로그 수집 설정 (예: ELK Stack, CloudWatch)
- 메트릭 정의 (CPU, 메모리, 요청 레이턴시, 에러율)
- 알림 규칙 설정 (임계값 초과 시 알림)
- 대시보드 구성 (Grafana, Datadog 등)

**중요**: COO는 "모니터링 설정 코드"를 작성하지만, "실시간 모니터링 운영"(온콜, 인시던트 응대)은 범위 밖.

#### 7.2.4 Runbook 작성
- 배포 절차 (사전 체크, 배포 명령, 사후 검증)
- 장애 대응 절차 (진단 방법, 롤백 방법)
- 스케일링 절차 (리소스 증가 시 수행 단계)
- 백업/복구 절차

#### 7.2.5 보안 설정 코드
- **네트워크 보안**: Security Group, Network ACL 설정
- **접근 제어**: IAM 정책, RBAC 설정
- **암호화 설정**: TLS 인증서, 키 로테이션
- **리소스 격리**: VPC, 서브넷 구성

**경계**: CSO가 보안 정책을 정의하고, COO가 정책을 코드로 구현.

### 7.3 하지 않는 것 (금지 사항)

| 금지 사항 | 담당 C-Level 또는 범위 밖 | 이유 |
|----------|------|-----|
| 실제 배포 명령 실행 | 범위 밖 (사용자 환경) | 플러그인은 코드만 작성, 실행은 사용자 책임 |
| 실시간 모니터링 및 온콜 | 범위 밖 | 플러그인은 설정만 제공 |
| 팀/조직 관리 | 범위 밖 | 조직 관리는 플러그인 범위 밖 |
| 기술 문서화 (아키텍처, API 가이드) | CTO | 기술 문서는 CTO 담당 |
| 코드 구현 또는 수정 | CTO | CTO가 비즈니스 로직 구현 |
| 제품 기획 | CPO | 제품 전략은 CPO |
| 보안 감시 및 감사 | CSO | CSO가 보안 정책 검증 |
| 마케팅 또는 비용 분석 | CBO | 비즈니스 전략은 CBO |

### 7.4 경계 케이스 (COO)

#### 케이스 1: 배포 자동화와 회귀 테스트
**상황**: CI/CD 파이프라인에 자동화된 테스트를 포함할지 말지 결정
- **COO의 책임**: 배포 전 자동화된 테스트 트리거 설정 (코드 작성)
- **CTO의 책임**: 테스트 자체(단위, 통합, E2E) 작성
- **경계**: COO는 "테스트 실행 명령"을 GitHub Actions에 추가, CTO는 테스트 코드 작성

#### 케이스 2: 모니터링 메트릭 정의
**상황**: 어떤 메트릭을 모니터링할지 결정
- **CTO의 책임**: 애플리케이션 성능 메트릭 제안 (응답 시간, 에러율)
- **CSO의 책임**: 보안 메트릭 제안 (로그인 실패, 비정상 트래픽)
- **CBO의 책임**: 비즈니스 메트릭 제안 (활성 사용자, 전환율)
- **COO의 책임**: 이 메트릭들을 수집/알림하는 코드 구성

#### 케이스 3: 스케일링 정책
**상황**: 트래픽 증가 시 자동 스케일링 정책 설정
- **COO의 책임**: Auto-scaling 정책 코드 작성 (CPU 50% 초과 시 인스턴스 추가 등)
- **CTO의 책임**: 애플리케이션이 수평 확장(horizontal scaling)에 대응 가능한지 확인
- **CBO의 책임**: 스케일링으로 인한 비용 증가 평가

#### 케이스 4: 재해 복구 (Disaster Recovery)
**상황**: 데이터 손실 또는 리전 장애 시 복구 전략
- **COO의 책임**: 백업/복구 스크립트 작성, RTO/RPO 정의
- **CTO의 책임**: 데이터 일관성, 상태 복구 로직
- **CBO의 책임**: 복구 비용 평가

#### 케이스 5: 환경 설정 및 비밀 관리
**상황**: API 키, 데이터베이스 비밀번호 등 민감 정보 관리
- **COO의 책임**: 비밀 관리 시스템(AWS Secrets Manager, HashiCorp Vault) 설정
- **CSO의 책임**: 비밀 로테이션, 접근 제어 정책 검증
- **CTO의 책임**: 애플리케이션이 비밀 시스템과 통합되도록 구현

#### 케이스 6: 배포 전략 (Blue-Green, Canary)
**상황**: 배포 중 다운타임이나 문제 최소화 전략 선택
- **COO의 책임**: Blue-Green, Canary, Rolling 배포 중 선택 및 자동화 코드 작성
- **CTO의 책임**: 각 배포 전략의 기술적 적합성 평가
- **CBO의 책임**: 배포 전략으로 인한 인프라 비용 영향 평가

## 8. 공통 원칙

### 원칙 1: Full PDCA 항상 적용

C-Level 간 위임 시 **항상 full PDCA**(Plan → Design → Do → QA → Report)를 돌린다.

```
Plan (계획):   "무엇을 할 것인가, 왜 하는가"
Design (설계): "어떻게 할 것인가, 기술 방안"
Do (실행):     "실제 구현/작성"
QA (검증):     "올바르게 되었는가, 문제는 없는가"
Report (보고): "결과를 종합 리포트, 다음 단계 제안"
```

AI 에이전트이므로 각 단계 수행 비용이 미미. 체계적 접근이 더 중요.

### 원칙 2: CEO 중앙 라우팅

**C-Level 간 직접 통신 없음.** 모든 위임과 핸드오프는 CEO를 경유한다.

```
예시: CTO가 보안 의심사항 발견
❌ 금지: CTO → CSO 직접 연락
✅ 허용: CTO → CEO 보고 → CEO가 CSO에게 위임
```

**목적**:
- 전체 컨텍스트 일관성 유지
- CEO의 조정 권한 보장
- 추적 가능성 (모든 소통의 기록)

### 원칙 3: CSO 최종 게이트

코드 수정이 발생하는 **모든** 시나리오에서 **CSO full PDCA**가 최종 코드 리뷰로 들어간다.

```
흐름:
1. CPO가 PRD 작성
2. CTO가 코드 구현 + 피어 리뷰
3. CSO가 보안 감시 (최종 게이트)
4. CSO 승인 후에만 배포 진행 (COO에게 전달 또는 사용자에게 제시)
```

**적용 범위**:
- 새로운 기능 구현
- 버그 수정 (보안 영향 가능성)
- 의존성 업데이트 (라이브러리 변경)
- 설정 변경 (보안 관련)

**CSO가 거부할 수 있는 이유**:
- 보안 취약점 발견
- 컴플라이언스 미충족
- 미흡한 보안 테스트

### 원칙 4: 수동/자동 모드

모든 시나리오는 두 가지 모드로 실행 가능:

| 모드 | 설명 | 언제 사용 |
|------|------|---------|
| **수동 모드** (기본) | 각 체크포인트마다 사용자 승인 필요 | 새로운 작업, 고위험 결정 |
| **자동 모드** (`--auto`) | CEO가 자율 실행, 체크포인트 없음 | 반복 작업, 저위험 결정 |

**중요**: 동작은 **완전히 동일**. 차이는 사용자 승인 여부만.

```
예시:
$ vais --task "새 기능 기획" [수동 모드]
> 계획 완료. 승인하시겠습니까? (Y/n)

$ vais --task "새 기능 기획" --auto [자동 모드]
> [승인 요청 없이 바로 실행]
```

### 원칙 5: 스코프 자율 규제

각 C-Level은 자신의 업무 범위 내에서 **자율적으로** 필요한 서브에이전트를 결정한다.

```
예시: CTO의 의사결정

"이 기능을 구현하려면 infra-architect, backend-dev, frontend-dev 필요"
→ CTO가 자율 결정, CEO에게 보고 후 진행

"보안 취약점 수정에는 security-audit, penetration-tester 필요"
→ CTO가 제안, CSO와 협의 (CEO 경유), 실행
```

**한계**:
- 자신의 업무 범위를 명백히 벗어나면 CEO 승인 필요
- 다른 C-Level의 책임 영역을 침범하면 안 됨

## 9. C-Level 간 핸드오프 규칙

### 9.1 핸드오프의 기본 구조

모든 핸드오프는 CEO를 경유한다. CEO는 이전 C-Level의 산출물을 받아서 다음 C-Level에 전달.

```
C-Level A의 PDCA 완료
        ↓
    CEO 검토
        ↓
C-Level B에 위임 (CEO가 컨텍스트 제공)
        ↓
C-Level B의 PDCA 실행
        ↓
    CEO에 결과 보고
```

### 9.2 주요 핸드오프 경로

#### 9.2.1 경로 1: CPO → CTO (기획 → 구현)

**CPO의 산출물 (입력)**:
- PRD (Functional & Non-Functional Requirements)
- 우선순위 순서
- 인수 기준 (Acceptance Criteria)
- UX 설계/프로토타입 (선택사항)

**CEO의 역할**:
- PRD 완성도 검증
- CTO의 기술적 우려사항 청취
- 명확하지 않은 부분 CPO에 재확인

**CTO의 책임**:
- PRD를 바탕으로 SDD (System Design Document) 작성
- 기술적 불가능성 조기 보고
- 설계 리뷰 (아키텍처 검증)

**반환 (산출물)**:
- 구현 완료 코드
- 테스트 결과 리포트

#### 9.2.2 경로 2: CTO → CSO (구현 → 보안 검증)

**CTO의 산출물 (입력)**:
- 완성된 코드 (GitHub PR 또는 브랜치)
- 테스트 리포트 (단위, 통합, E2E)
- 기술 문서 (API, 아키텍처)

**CEO의 역할**:
- CTO의 작업 완료도 검증
- CSO의 우려사항 청취
- 재작업 필요 시 조정

**CSO의 책임**:
- 전체 코드 보안 감시
- 컴플라이언스 검증
- 테스트 커버리지 확인

**회신 (산출물)**:
- 보안 감사 리포트
- 승인 또는 거부
- 개선 사항 목록 (거부 시)

#### 9.2.3 경로 3: CSO → COO (승인 → 배포 준비)

**CSO의 산출물 (입력)**:
- 보안 승인 (signed-off)
- 배포 전 체크리스트

**CEO의 역할**:
- CSO의 승인 확인
- COO에게 배포 설정 작성 지시

**COO의 책임**:
- CI/CD 파이프라인 작성
- 배포 스크립트 작성
- Runbook 작성

**반환 (산출물)**:
- 배포 자동화 코드
- 모니터링 설정
- 장애 대응 절차

#### 9.2.4 경로 4: 사업 결정 (CPO ↔ CBO ↔ CTO)

**상황**: 새 기능이 기술적으로 비싸거나 마케팅적으로 중요할 때

**흐름**:
1. CPO가 기능 기획
2. CBO가 시장 가치와 비용 분석
3. CTO가 기술적 투입 시간 추정
4. CEO가 투입(비용) vs. 효과(수익)를 종합 판단

**핵심**: 어느 C-Level도 일방적 결정 불가. 모든 입력이 필요.

### 9.3 대기(Blocking) 및 우선순위

#### 9.3.1 대기 사항 (Blocker)

```
CPO의 작업이 완료될 때까지 CTO는 대기
CTO의 작업이 완료될 때까지 CSO는 대기
CSO의 승인 전까지 COO는 배포 설정 작성 불가
```

**예외**: CTO가 선제적으로 아키텍처를 설계하거나 인프라를 준비할 수 있음. 하지만 CPO 승인 없이 본격 구현은 금지.

#### 9.3.2 우선순위 변경

사용자 요청이 변경되거나 우선순위가 바뀌면:

1. **CEO가 판단**: 우선순위 변경의 정당성 평가
2. **영향받는 C-Level들에게 통보**:
   - 이미 진행 중인 작업의 취소/연기
   - 새 작업의 추가
3. **일정 재조정**: CTO와 COO가 새 일정 수립

## 10. 경계 케이스 종합 (모든 C-Level)

### 10.1 모호한 영역의 최종 판정자

| 모호한 영역 | 최종 판정자 | 근거 |
|-----------|----------|-----|
| 기술 선택 (라이브러리, 프레임워크) | CTO (보안 우려 시 CSO 동의 필수) | 기술 리더십 |
| 기능 스코프 (포함 vs. 미포함) | CPO (기술 불가능 시 CEO 중재) | 제품 전략 |
| 보안 정책 수준 (엄격 vs. 유연) | CSO | 독립적 보안 검증 |
| 비용 투입 (고비용 vs. 저비용) | CBO (시장 가치와 비용 균형) | 재무 책임 |
| 마케팅 메시지 | CBO (제품 기능과 일관성 확인은 CPO) | 사업 전략 |
| 배포 전략 (Blue-Green vs. Canary) | COO (성능/비용 영향은 CTO/CBO 자문) | 운영 책임 |

### 10.2 이해가 충돌하는 경우의 해결 절차

```
단계 1: 양쪽 C-Level의 입장 청취 (CEO가 진행)
단계 2: 타협안 모색 (예: 부분 구현, 단계적 롤아웃)
단계 3: 타협 불가능하면 CEO 최종 결정
단계 4: 결정 사항을 모든 관련자에게 통보 및 문서화
```

**예시: 보안과 사용성 충돌**
```
CSO: "2FA 필수 적용 필요"
CPO: "사용성 저하로 고객 이탈 우려"

CEO의 조정:
1. 특정 사용자(관리자)는 2FA 필수
2. 일반 사용자는 2FA 선택사항
3. 3개월 후 재평가
```

## 11. 시나리오별 역할 분담 (사례 연구)

### 시나리오 1: 새 기능 요청 (완전한 흐름)

**사용자 요청**: "결제 기능 추가 (신용카드 결제)"

```
CEO:    요청 분석 → CPO, CTO, CSO, CBO 필요 판단
↓
CPO:    시장조사 → PRD 작성 (결제 플로우, 보안 요구사항)
↓
CEO:    PRD 검증 → CTO에 전달
↓
CTO:    SDD 작성 → 결제 API 선택, DB 스키마 설계, 결제 모듈 구현
↓
CSO:    코드 검증 → PCI-DSS 준수 확인, 카드 정보 암호화 검증
↓
CBO:    가격/ROI 분석 → 결제 수수료 비용, 예상 수익 계산
↓
CEO:    종합 리포트 생성 → 완료 및 배포 승인
↓
COO:    배포 설정 → CI/CD, 모니터링, Runbook 작성
↓
(배포 및 런칭)
```

### 시나리오 2: 보안 취약점 발견 (긴급 핸드오프)

**상황**: CSO가 프로덕션 코드에서 SQL Injection 취약점 발견

```
CSO:    취약점 보고 → CEO에 긴급 알림
↓
CEO:    심각도 평가 (치명적 = 즉시 대응)
↓
CTO:    수정 구현 → 임시 완화 조치 + 근본 해결책 개발
↓
CSO:    재검증 → 수정 확인
↓
CEO:    배포 승인
↓
COO:    긴급 배포 스크립트 실행
↓
(모니터링 및 회고)
```

### 시나리오 3: 기술과 비용의 트레이드오프

**상황**: CTO가 "캐싱 인프라 추가로 성능 10배 개선 가능. 비용은 월 $5,000"이라고 제안

```
CTO:    기술 제안 → CEO에 보고
↓
CBO:    비용-편익 분석 → 월 $5,000 비용, 예상 수익 증가 분석
↓
CPO:    사용자 경험 영향 → 성능 개선의 마케팅 가치 평가
↓
CEO:    최종 결정 → "실행" 또는 "보류"
```

## 14. 용어 정의

| 용어 | 정의 |
|------|------|
| **PDCA** | Plan(계획) → Design(설계) → Do(실행) → QA(검증) → Report(보고) 사이클 |
| **PRD** | Product Requirements Document. 제품 기능, 요구사항, 인수 기준 명시 |
| **SDD** | System Design Document. 기술 아키텍처, DB 스키마, API 설계 등 |
| **핸드오프** | 한 C-Level의 작업 완료 후 다음 C-Level에 산출물 전달 |
| **금지 사항** | 각 C-Level이 명확히 수행하면 안 되는 업무 |
| **경계 케이스** | 두 C-Level 간 책임이 모호한 상황 |
| **CVSS** | Common Vulnerability Scoring System. 보안 취약점 심각도 평가 (0-10, 높을수록 심각) |
| **Unit Economics** | 고객당 평균 수익(ARPU), 고객 획득 비용(CAC) 등 기본 재무 지표 |
| **RTO** | Recovery Time Objective. 시스템 복구에 소요 가능한 최대 시간 |
| **RPO** | Recovery Point Objective. 데이터 손실 허용 최대 범위 |
